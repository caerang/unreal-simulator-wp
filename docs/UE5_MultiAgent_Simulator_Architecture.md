# 🚀 국방/항공 다중 에이전트 강화학습 시뮬레이터 구축 문서

**최종 업데이트:** 2026년 6월 
**아키텍처 구조:** 언리얼 엔진 5(UE5) 기반 데이터 주도형 다중 환경 스폰 및 파이썬 gRPC 원격 통신 파이프라인

## 1. 프로젝트 개요
본 프로젝트는 국방/항공 시뮬레이션(드론, 제트기, 함선 등)을 위한 대규모 강화학습(RL) 환경을 언리얼 엔진 내에 구축하는 것을 목표로 합니다. 외부 임무 데이터(CSV)를 기반으로 다수의 시나리오 환경을 동적으로 생성하며, 파이썬 기반의 강화학습 모델과 네트워크 통신을 통해 에이전트의 관측(State)과 행동(Action)을 실시간으로 주고받는 확장형 프레임워크입니다.

## 2. 현재까지 구현 완료된 사항 (Implementation)

### 2.1. 데이터 주도형 동적 스폰 시스템 구축
* **외부 데이터 연동:** 엑셀(CSV) 형식의 시나리오 데이터(`Scenario_List`)를 언리얼 엔진의 데이터 테이블로 임포트 성공.
* **자동화된 스폰 로직:** 레벨 블루프린트에서 `For Each Loop`를 활용해 CSV의 행(Row) 데이터를 읽고, 각 좌표에 독립적인 전투기 객체(`BP_Fighter`)를 동적 스폰하도록 구현.
* **콜리전 이슈 해결:** 생성 시 바닥과 충돌하여 스폰이 누락되는 현상을 막기 위해, Z축 좌표 상향 조정 및 `Always Spawn, Ignore Collisions` 옵션 적용 완료.

### 2.2. 인공지능(AI) 두뇌 탑재 및 통신 파이프라인
* **Pawn 클래스 전환:** 큐브를 단순 사물(Actor)에서 AI가 탑승할 수 있는 `Pawn`으로 업그레이드 완료.
* **AI 3종 세트 연동:** `AIC_Fighter`(조종사), `BB_Fighter`(블랙보드/기억력), `BT_Fighter`(행위 트리/판단력) 생성 및 Pawn에 자동 빙의(Auto Possess AI) 설정 완료.

### 2.3. 비동기 타이밍(Race Condition) 에러 해결
* **문제:** AI가 큐브에 탑승하기도 전에 뇌(블랙보드)를 호출하여 발생하는 `Accessed None` 치명적 오류 발생.
* **설계 개선(Best Practice 적용):** 불안정한 `Delay` 노드를 폐기하고, 조종사(`AIC_Fighter`) 중심의 명시적 초기화 아키텍처로 전면 개편.
  1. `Event OnPossess` (빙의 확인)
  2. `Use Blackboard` (뇌의 명시적 생성 및 할당)
  3. `Set Value as Name` (비행기 주머니에서 임무 데이터를 꺼내 뇌에 기록)
  4. `Run Behavior Tree` (모든 데이터 준비 완료 후 판단 시작)

### 2.4. 규칙 기반(Rule-based) 행동 분기 구현
* **행위 트리(BT) 분기 로직:** 블랙보드에 기록된 `MissionTarget` 변수 값("Drone", "Jet", "Ship")에 따라 행동을 달리하는 `Selector` 및 3개의 `Sequence` 분기점 구축 완료.
* **트리 생존용 Fallback:** 조건 불일치 시 트리가 즉시 강제 종료(Instant Death)되는 것을 방지하기 위해 `Wait 500s` 안전망 노드 적용, 디버그 목록 실시간 모니터링 성공.

### 2.5. C++ 모듈 마이그레이션 및 gRPC 연동 완료 (2026-06-13)
* **C++ 프로젝트 하이브리드화:** 순수 블루프린트 프로젝트에서 C++ 모듈 컴파일이 가능하도록 프로젝트 구조(`Source`, `.Target.cs`, `.Build.cs`)를 설계 및 UBT 연동 완료.
* **Unreal Engine 5.7 빌드 호환성 대응:** Installed Engine의 공유 빌드 환경 규격과의 속성 충돌을 해결하기 위해 `DefaultBuildSettings = BuildSettingsVersion.V6` 및 `CppStandard = CppStandardVersion.Cpp20`을 타겟 설정에 적용하여 컴파일을 완수함.
* **gRPC 라이브러리 및 플러그인 빌드:** `Source/ThirdParty`에 사전 빌드된 gRPC C++ 라이브러리 바이너리를 적재하고, **TurboLink** 플러그인을 프로젝트 내 빌드 트리에 포함시켜 `UnrealEditor-TurboLinkGrpc.dll` 빌드 성공.
* **gRPC 코드 제네레이션:** `FighterRL.proto` 스키마로부터 언리얼 C++ 클래스 및 블루프린트 비동기 호출 노드 코드를 자동 생성하여 플러그인 소스 트리에 병합 완료.
* **Python gRPC 서버 완비:** 파이썬 가상환경 구축 및 `grpcio` 설치를 완료하고, 언리얼의 관측 데이터(`State`)를 받아 더미 기동 제어값(`Action`)을 응답하는 [python_agent/server.py](file:///D:/workspace/simulator_wp/python_agent/server.py)를 구축함.

## 3. 설계 검토 및 결정 사항 (Architecture Decisions)

### 3.1. 대규모 확장(256개+)을 위한 '공간 분할 병렬 처리' 도입
단일 엔진 프로세스 내에서 256개의 개별 학습 환경을 구동하기 위해 아키텍처를 다음과 같이 설계하기로 합의함.
* **지형 기반 오프셋(Geographic Offset):** 시나리오별 거점(예: 강릉, 평양, 대구)의 절대 좌표를 중심으로, 소속 에이전트들을 거점 주변에 스폰하여 논리적/물리적 환경 격리.
* **LWC 및 단일 틱 공유:** 언리얼 엔진 5의 거대 월드 좌표(LWC)를 활용하여 오차를 없애고, 256개 환경이 완전히 동일한 시간(Time Step)을 공유하여 파이썬 측 데이터 노이즈 원천 차단.
* **최적화 정책:** 학습 시에는 `-nullrhi` 커맨드를 활용한 헤드리스(Headless) 모드로 구동하여 GPU 렌더링 부하를 제거하고 연산 속도를 극대화.

### 3.2. 언리얼-파이썬 간 통신 규약: gRPC 채택
언리얼 공식 플러그인 대신, 산업 표준인 **gRPC**를 채택함.
* **결정 사유:** 클라우드 GPU 서버와의 원격 분산 학습, 파이썬 프레임워크(PyTorch, RLlib 등)에 대한 높은 자유도, 256개 이상 스케일업 시 서버/클라이언트의 명확한 아키텍처 분리.

### 3.3. 프로토콜 버퍼(.proto) 초안 설계 완료
강화학습 통신의 핵심인 State/Action 주고받기를 위한 `FighterRL.proto` 규약 설계.
* **AgentState (UE ➡️ Python):** 환경 ID, 에이전트 종류, X/Y/Z 3차원 좌표, 방향, 타겟 거리 등 관측 정보.
* **AgentAction (Python ➡️ UE):** 엔진 추진력(Thrust), 조향(Steering) 등 파이썬이 계산한 제어 수치.
